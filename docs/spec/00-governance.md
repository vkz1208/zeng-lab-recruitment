## 0.1 状态定义

- 待审核：已写入文档，但尚未获得项目拥有者确认，不作为最终实施依据。
- 已采纳：已获得项目拥有者确认，可作为实施依据。
- 已失效：原内容不再适用，但不得删除；必须保留失效时间、失效原因和替代内容。
- 已替代：原内容仍可追溯，但后续实施以新的已采纳内容为准。

## 0.2 修改规则

- 每次修改 SPEC 前，必须先提出变更草案，说明变更范围、原因、影响模块和预期结果。
- SPEC 的新增、修改、失效标注，必须经过项目拥有者同意后才能生效。
- 不得直接删除或覆盖历史内容。若内容不再适用，应标注“已失效”、失效时间、失效原因和替代内容。
- 新内容必须标注采纳时间和采纳状态。
- 功能实现必须优先遵循“已采纳”内容；“待审核”内容仅作为讨论与规划依据。

## 0.3 多文档维护规则

- `docs/SPEC.md` 是总入口，不承载完整需求细节。
- 具体需求应写入 `docs/spec/` 下的专题文档。
- 新增专题文档时，必须在总入口目录中登记。
- 跨专题变更必须同时更新受影响文档的变更记录。
- 若不同专题文档出现冲突，以更新日期更晚且状态为“已采纳”的内容为准。
